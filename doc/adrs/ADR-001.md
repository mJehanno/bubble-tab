# ADR-001: TabModel architecture — lazy init, deferred construction, bug fixes, and theming

- **Status:** Accepted
- **Date:** 29-06-2026
- **Author:** mathob.jehanno@ovhcloud.com

## Context

The `bubble-tab` library provides a reusable `TabModel` for Bubble Tea v2
(`charm.land/bubbletea/v2`) TUI applications. No tab component exists for
Bubble Tea v2 in the official `bubbles` set or in community libraries (all
known implementations target v1), so this library fills a genuine gap.

Three requirements drive the design:

1. **Navigation** — keyboard and mouse navigation between tabs, with a
   configurable keymap, number-key jumps (1–9), and correct back-cycling
   (Shift+Tab).
2. **Lazy loading** — a tab's body must not be initialised or rendered until
   the tab is first activated; state must survive subsequent tab switches
   without re-running `Init`.
3. **Themeable styling** — the component must support rich colour themes
   matching community palettes (Gruvbox, TokyoNight, Catppuccin) with both
   dark and light variants.

The existing partial implementation had several correctness bugs:
- Child `Init` was called on every tab switch instead of once.
- Messages were never forwarded to the active child's `Update`, so child
  state was always frozen.
- `WindowSizeMsg` was never broadcast to off-screen tabs, leaving them
  incorrectly sized on first activation.
- `moveBackward` used a `uint64` index, causing underflow/wrap-around when
  decrementing from zero.
- The disabled-tab skipping loop could infinite-loop when all tabs are
  disabled.
- Theme struct field names were typo'd (`Secundary`, `Tercary`).
- Body styling was applied to a `lipgloss.Style` that was never used in
  rendering.

## Decision

The **A+B hybrid** approach was chosen:

- Each `Tab` tracks an `initialized` flag. `Init` runs exactly once on first
  activation; the live child model is cached and persisted thereafter, so
  state survives across tab switches.
- An optional `WithBodyFunc(func() tea.Model)` constructor is added so that
  the model is not even allocated until first activation, covering expensive
  bodies. The existing `WithBody(tea.Model)` is retained for the common
  cheap/eager-allocation case.
- Messages are forwarded to the active child's `Update` and the returned
  model is persisted. `WindowSizeMsg` is broadcast to all tabs.
- Navigation correctness fixes: `uint64` underflow replaced with a safe
  signed comparison; disabled-tab loop made termination-safe; Alt+Tab
  corrected to Shift+Tab; number-key (1–9) jumps added; clickable tab
  headers added via Bubble Tea v2 `View.OnMouse` mouse support; keymap
  exposed as a configurable `key.Binding` struct.
- Theming: a public `Theme` struct with corrected field names (`Secondary`,
  `Tertiary`) carries dark and light colour variants; a derived `Styles`
  struct holds precomputed `lipgloss` styles per state/part; body styling is
  wired into rendering. Prebuilt adapters are provided: `themes.Gruvbox`,
  `themes.TokyoNight`, `themes.Catppuccin`, each with dark and light variants
  where applicable.

## Considered Options

- **[A+B hybrid]** — the chosen option: lazy-init-once with cached live
  models, plus optional deferred construction via `WithBodyFunc`, plus full
  bug fixes and rich theming with prebuilt adapters.
- **[B — factory-only lazy]** — tab bodies constructed exclusively via
  factory closures; no eager `WithBody`. Zero allocation for never-visited
  tabs, but the closure-based API is less ergonomic for the common cheap
  case and harder to pre-seed initial state. Rejected as the sole approach;
  folded in as the optional `WithBodyFunc` path.
- **[C — minimal correctness fix]** — fix only the init-on-every-switch and
  message-forwarding bugs plus the field-name typos; keep theming primitive
  with no prebuilt theme adapters. Cheapest and smallest surface, but
  under-delivers on the explicitly required Gruvbox/TokyoNight/Catppuccin
  theme support.

## Pros

- Satisfies all three stated requirements: navigation (including mouse),
  true lazy loading, and rich themeable styling.
- Correct lazy semantics: child state persists across tab switches; `Init`
  runs exactly once per tab.
- Fixes all identified correctness bugs (frozen children, `uint64` underflow,
  infinite disabled-tab loop, unsized off-screen tabs).
- Ergonomic for the common case (`WithBody`) while also accommodating
  expensive bodies (`WithBodyFunc`).
- Prebuilt theme adapters match the concrete palettes requested, with
  dark/light variants.

## Cons

- Larger surface and more upfront design than the minimal fix (Option C).
- Two body-provisioning paths (`WithBody` and `WithBodyFunc`) to document
  and maintain.
- Mouse support and number-key jumps add navigation complexity and
  maintenance surface.
- Child models must tolerate being constructed but not yet `Init`'d — this
  is the standard Bubble Tea contract and is considered low risk.

## Reason

The A+B hybrid is the only option that fully delivers all three explicit
requirements — including mouse support noted as desirable — while also
resolving the latent correctness bugs in the existing partial implementation.
Option B alone sacrifices ergonomics in the common case. Option C fails to
deliver the theming requirement. The additional maintenance cost of two
provisioning paths and richer navigation is justified by filling a genuine
Bubble Tea v2 gap with a component that is both correct and immediately
useful to the broader community.
