# ADR-002: Tab header spacing and style customisation

- **Status:** Accepted
- **Date:** 29-06-2026
- **Author:** mathob.jehanno@ovhcloud.com

## Context

[ADR-001](ADR-001.md) established the `TabModel` architecture, including a
`Theme`/`Styles` pair where `Styles` is derived from `Theme` internally and
never exposed for external override. As a consequence, library users had no
public way to control the padding or margin of the rendered tab headers, and no
way to substitute entirely custom `lipgloss` styles. A user reported this gap
and requested a first-class customisation API.

Two distinct needs were identified:

1. **Common case** — the user is happy with the theme's colours but wants to
   adjust spacing (padding, margin) on the header elements.
2. **Power-user case** — the user wants complete control and prefers to supply
   their own `Styles` struct rather than layering on top of a theme.

An additional sub-problem arose from Go's zero-value ambiguity: if spacing
options default to zero, `WithPadding(0, 0)` is indistinguishable from "not
set", which would silently discard an explicit user intent to collapse spacing
to zero.

A further consequence worth recording is the impact on mouse hit-testing.
`TabModel.View()` determines the clickable spans for each header by measuring
the rendered (already-spaced) header widths via `lipgloss.Width`. The
`OnMouse` handler therefore sees the same widths used for rendering, making
the hit-test spans self-consistent by construction. No extra offset
bookkeeping is required when spacing changes.

## Decision

Two complementary opt-in mechanisms are provided, both implemented in
`main.go`:

1. **`WithStyles(theme.Styles)`** — replaces the theme-derived `Styles`
   wholesale. The supplied struct is used as-is; no merging with theme values
   occurs.
2. **`WithPadding(x, y int)` and `WithMargin(x, y int)`** — convenience
   functional options on `TabModel`. `x` is the horizontal (left/right)
   value and `y` is the vertical (top/bottom) value, matching the CSS
   shorthand order used by lipgloss (`vertical, horizontal`). Both options
   are applied by `applyHeaderSpacing()` to the three header styles only:
   `Active`, `Inactive`, and `Disabled`. The tab body is unaffected.

Zero-value-with-flag semantics are used: separate `hasPadding`,
`hasMargin`, and `hasCustomStyles` boolean fields on `TabModel` ensure that
an explicit call to, for example, `WithPadding(0, 0)` is honoured as a
deliberate instruction to collapse spacing, rather than treated as "unset".

Spacing applies to the three header styles only. The tab body is excluded.

## Considered Options

- **[WithStyles + WithPadding/WithMargin]** — the chosen option: two
  complementary APIs, each targeting a different level of control.
- **[WithPadding/WithMargin only]** — spacing convenience helpers without
  the wholesale-replacement escape hatch. Covers the common case but leaves
  power users without a full-control path.
- **[Padding/margin fields on theme.Theme]** — add spacing directly to the
  `Theme` struct. Rejected: spacing is an orthogonal concern to colour/theme;
  coupling them conflates two responsibilities and is least flexible (a user
  who wants different spacing for an existing prebuilt theme would need to
  clone the whole `Theme`).
- **[Merging per-field onto the theme in WithStyles]** — a partial-override
  design where only non-zero fields in the supplied `Styles` replace theme
  values. Declined by the user in favour of a simpler wholesale-replacement
  semantics.
- **[Extending spacing to the tab body]** — applying `WithPadding`/
  `WithMargin` to the body style as well as the three header styles.
  Declined by the user; header-only is sufficient and keeps the body layout
  under the application's control.

## Pros

- Covers both the common case (spacing tweak) and the power-user case (full
  style replacement) without API bloat.
- `WithPadding`/`WithMargin` are ergonomic and self-documenting; no need to
  construct a `Styles` struct for a simple spacing adjustment.
- Zero-value-with-flag semantics mean `WithPadding(0, 0)` behaves
  predictably — the explicit zero is always honoured.
- Mouse hit-testing remains correct without any extra offset bookkeeping,
  because `View()` derives all spans from `lipgloss.Width` on the
  already-rendered (already-spaced) headers.
- `WithStyles` gives an unambiguous escape hatch for users who need colours,
  borders, and spacing all customised together.

## Cons

- The `hasPadding`/`hasMargin`/`hasCustomStyles` flag fields add a small
  amount of bookkeeping state to `TabModel`.
- Two separate customisation paths (`WithStyles` and the spacing helpers)
  must both be documented; a user who sets both must understand that
  `WithStyles` replaces the base, after which `WithPadding`/`WithMargin`
  are still applied on top via `applyHeaderSpacing()`.
- `WithStyles` replaces wholesale, so a user who wants to change one colour
  must supply a complete `Styles` value — no per-field merge is provided.

## Reason

The pairing of `WithStyles` and `WithPadding`/`WithMargin` satisfies both
stated use cases with minimal API surface. Spacing-only adjustments are the
overwhelmingly common request, and the two dedicated helpers cover that
cleanly without requiring the user to know the internals of `Styles`. The
wholesale-replacement design of `WithStyles` was preferred over a
per-field-merge design because it is simpler to reason about — the caller
supplies exactly what will be used. Placing spacing on `Theme` was rejected
because it would conflate colour and layout concerns in the same struct,
making existing prebuilt theme adapters harder to reuse unchanged.
